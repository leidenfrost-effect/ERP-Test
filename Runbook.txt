Hybrid ERP Uygulamanız İçin Codex/CLI Runbook’u
Kurumsal ürünleşmeye giden yolda otomatik güvenlik/teknik tarama → geliştirici görevleri üretme → SSO/OAuth2+PKCE, RBAC, OpenAPI, audit logging, SBOM, OpenTelemetry, offline caching adımlarını kopyala‑yapıştır çalıştırılabilir prompt/komut bloklarıyla uçtan uca uygular.
Yönetici özeti
Ekran görüntüleriniz iki kritik olgunluk boşluğunu net gösteriyor: (i) istemci tarafında API Base URL + opsiyonel Bearer token ile “debug/PoC” seviyesinde oturum yaklaşımı, (ii) “Ürün Kataloğu” ekranında 404 Not Found → JSON parse (FormatException) hatası; yani istemcinin non‑JSON yanıtı JSON sanıp parse etmesi ve/veya endpoint sözleşmesinin belirsizliği. Bu tip hataları kalıcı olarak çözmenin en hızlı yolu OpenAPI sözleşmesi (design/contract-first) + otomatik contract test + typed client kombinasyonudur. OpenAPI, HTTP API’lerini dil bağımsız şekilde tanımlayan standarttır ve spec’in kendisi, bu tanımın kod üretimi ve test gibi kullanım alanlarını hedeflediğini açıkça belirtir. [1]
Kimlik tarafında, mobil/native uygulamalar için yetkilendirme isteklerinin harici kullanıcı aracısı (sistem tarayıcısı) üzerinden yapılması (embedded webview yerine) RFC 8252’nin önerdiği “best practice”tir. [2] Authorization Code akışının mobilde güvenli uygulanması için PKCE (RFC 7636) kullanımı, “authorization code interception” riskine karşı temel mitigasyondur. [3]
Kurumsal ürünleşmeyi “süreç” tarafında kalıcı kılmak için NIST SSDF (SP 800‑218) pratiklerini sprint kalite kapısı olarak bağlamak; mobil güvenlikte OWASP MASVS, backend/web’de OWASP ASVS ve API’lerde OWASP API Security Top 10 (özellikle Broken Object Level Authorization / IDOR) kontrollerini otomasyona dönüştürmek gerekir. [4]
Kapsam, ekranlardan gözlemler ve belirsizler
Aşağıdaki iki ekran, runbook’un neden bu sırayla tasarlandığını gösteriyor.
Giriş ekranı: API Base URL + opsiyonel Bearer Token

Bu giriş yaklaşımı geliştirmede hızlıdır; fakat üretimde kullanıcıya URL/token verdirme, yanlış ortam/zararlı endpoint, token sızıntısı, KVKK kapsamındaki erişim loglarının güvenilmezliği gibi riskler doğurur. KVKK’nın amacı kişisel verileri korumaktır; İK modülü gibi alanlar bu kapsamı doğrudan tetikler. [5]
Ürün Kataloğu: 404 Not Found → FormatException

Bu hata tipi genelde şu kök sebeplere işaret eder: (i) endpoint gerçekten yok / yanlış base path, (ii) reverse proxy route, (iii) auth middleware 404/HTML döndürüyor, (iv) istemci “content-type” kontrol etmeden JSON decode ediyor. Bu runbook önce sözleşme ve test kurar, sonra kimlik + RBAC + audit ekler.

Örnek Flutter OAuth2+PKCE istemci iskeleti (temsilî – repo’ya prompt ile entegre ettirin; gerçek paket/ayarlar stack’e göre değişir):
// PSEUDO-CODE: OAuth2 Authorization Code + PKCE (native app best practice: system browser) [17]
final authConfig = OAuthConfig(
  issuer: "https://idp.example.com/realms/erp",
  clientId: "nova-erp-mobile",
  redirectUri: "com.nova.erp://auth/callback",
  scopes: ["openid", "profile", "email", "erp.api"],
);

final result = await oauth.authorizeAndExchangeCode(
  usePkce: true,
  preferEphemeralSession: false,
);

await secureStorage.write("access_token", result.accessToken);
await secureStorage.write("refresh_token", result.refreshToken);

// API call wrapper
Future<Response> getJson(String path) async {
  final token = await secureStorage.read("access_token");
  final res = await http.get(
    Uri.parse("$baseUrl$path"),
    headers: {"Authorization": "Bearer $token", "Accept": "application/json"},
  );
  if (res.statusCode < 200 || res.statusCode >= 300) {
    // content-type kontrolü: 404/HTML gibi yanıtları JSON decode etmeyin
    throw ApiException.fromHttp(res);
  }
  return res;
}

OpenTelemetry enstrümantasyonu ve korelasyon
Amaç: istemci‑API‑DB zincirinde correlation/trace id ile hata analizi; audit event’leriyle bağ.
Beklenen çıktı:
backend OTel SDK init, OTLP exporter konfig, traceparent propagation, correlation id middleware; opsiyonel collector config.
Ön koşullar: gözlemlenebilirlik backend’i unspecified; OTLP collector önerilir.
Güvenlik/uyumluluk: OpenTelemetry, telemetry (traces/metrics/logs) üretme‑toplama‑export etmeye yönelik vendor‑neutral çerçevedir. [26] Enstrümantasyon kavramı ve yöntemleri OTel dokümanlarında tanımlanır. [27] Trace context propagation için W3C Trace Context standardı (traceparent/tracestate) temel HTTP header formatını tanımlar. [28] KVKK/GDPR: telemetry’de PII taşınmamalı; kullanıcı id yerine pseudonymous id veya hash tercih edilmeli. [9]
[CODEX/CHATGPT PROMPT — OpenTelemetry + correlation + audit link]
Amaç: Backend'e OpenTelemetry tracing ekle ve audit logging ile ilişkilendir.

İş:
1) Backend stack'ini algıla (Node/.NET/Java/Python):
   - Uygun OpenTelemetry SDK init ekle
   - OTLP exporter (endpoint env var)
2) Middleware:
   - Her request için correlation_id üret (varsa X-Correlation-Id'yi kabul et)
   - W3C traceparent header'ını koru/forward et
3) Audit event içine correlation_id + trace_id yaz
4) docs/observability.md:
   - local dev: OTEL_EXPORTER_OTLP_ENDPOINT örneği
   - prod: sampling, PII policy (kvkk/gdpr)
5) Test:
   - correlation id header geri dönüyor mu?
   - audit event trace id içeriyor mu?

Çıktı:
- backend OTel init + middleware + test
- docs/observability.md
Offline cache ve senkronizasyon kuyruğu
Amaç: depo/stok/operasyon gibi modüllerde bağlantı dalgalanmalarında veri kaybı olmaması; “eventual sync”.
Beklenen çıktı:
mobilde local DB/cache katmanı, sync queue, conflict resolution politikası, offline mod testleri.
Ön koşullar: mobil stack unspecified (Flutter/RN/Ionic).
Güvenlik/uyumluluk: mobilde yerel depolama hassas olabilir; OWASP MASVS mobil güvenlik doğrulama standardı olarak kullanılmalıdır. [29] KVKK/GDPR: cihazda tutulan PII minimize edilmeli; at-rest encryption + remote wipe politikası düşünülmeli. [9]
[CODEX/CHATGPT PROMPT — Offline-first veri katmanı + sync]
Amaç: Mobil istemcide offline cache + sync queue uygulamak.

İş:
1) Mobil stack algıla:
   - Flutter ise: SQLite tabanlı bir persistence katmanı (repository pattern)
   - React Native ise: SQLite/WatermelonDB benzeri
   - Ionic ise: IndexedDB/SQLite wrapper
2) Tasarım:
   - Entity cache: products, orders, inventory snapshots (TTL + ETag/If-None-Match opsiyonu)
   - Write operations: "outbox" tablosu (queued mutations)
   - Sync engine:
     - online olunca FIFO replay
     - 409 conflict için basit strateji: server-wins + kullanıcıya uyarı
3) API:
   - Mutations idempotency key header ekle (tekrar denemelerde çift yazmasın)
4) Test:
   - offline iken create/update → outbox’a düşüyor mu?
   - online olunca sync oluyor mu?
   - 404 / non-json durumda UI düzgün hata veriyor mu? (FormatException fix regression)

Çıktı:
- mobile persistence + sync code
- tests
- docs/offline-sync.md
